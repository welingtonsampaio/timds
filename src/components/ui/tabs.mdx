# Tabs ┬Ę `Stable`

Switches between sibling panels of content within the same context, one visible at a time.

## Overview

Use Tabs to organize related content into panels the user can switch between without
leaving the page. It is built on Radix Tabs, so it ships the correct `tablist`/`tab`/
`tabpanel` roles and arrow-key navigation. The `TabsList` offers four visual variants
(`default`, `line`, `pill`, `card`), plus `size`, `centered` and `tabPlacement`, while
`TabsTrigger` adds `closable`/`onClose` for editable tab strips.

## Anatomy

- **Tabs** ŌĆö the root; holds the selected value and `tabPlacement`.
- **TabsList** ŌĆö the `role="tablist"` rail carrying the `variant`, `size` and `centered`
  layout.
- **TabsTrigger** ŌĆö a single `role="tab"`; with `closable` it shows a close affordance.
- **TabsContent** ŌĆö the `role="tabpanel"` shown for the active tab.

## When to use

- Use to split related content into a few peer sections within one view.
- Use `value` + `onValueChange` to drive the active tab from state, or `defaultValue`
  to seed it.
- Use `closable` tabs for an editable, dynamic set (e.g. open documents).

## When not to use

- Avoid for sequential steps that must be completed in order ŌĆö use a stepper/wizard.
- Avoid for navigation between distinct pages or routes ŌĆö use links/navigation.
- Avoid hiding critical content the user always needs visible behind a tab.

## Behavior & interaction

These behaviors are guarded by interaction tests in `tabs.stories.tsx`:

- Clicking a tab activates it and swaps the visible panel ŌĆö `SwitchesTab`.
- Arrow keys move focus and selection between tabs ŌĆö `NavigatesWithArrows`.
- The component exposes `tablist`/`tab`/`tabpanel` roles with `aria-selected` on the
  active tab ŌĆö `ExposesTabRoles`.
- A disabled tab cannot be selected, even when clicked ŌĆö `DisabledNotSelectable`.
- Closable tabs can be closed by the (x) affordance or by Delete/Backspace on the focused
  tab; the editable demo adds and removes tabs ŌĆö `Editable`.

## Code examples

```tsx
import { Tabs, TabsContent, TabsList, TabsTrigger } from 'timds'

<Tabs defaultValue="home">
  <TabsList variant="pill">
    <TabsTrigger value="home">Home</TabsTrigger>
    <TabsTrigger value="products">Products</TabsTrigger>
  </TabsList>
  <TabsContent value="home">Home panel</TabsContent>
  <TabsContent value="products">Products panel</TabsContent>
</Tabs>

// Editable tabs
<TabsTrigger value="1" closable onClose={() => remove('1')}>
  Tab 1
</TabsTrigger>
```

## Content & UX writing

- **Tab labels** ŌĆö short noun phrases in sentence case ("Overview", "Billing"); one or
  two words where possible.
- **No terminal punctuation** in tab labels.
- **Stable order** ŌĆö keep tab order stable across sessions; don't reorder by recency.
- **Close affordance** ŌĆö only mark tabs `closable` when removal is genuinely supported.

## Accessibility

- Exposes `role="tablist"` on the list, `role="tab"` on each trigger and
  `role="tabpanel"` on the content; the active tab carries `aria-selected="true"` ŌĆö
  verified by `ExposesTabRoles`.
- Keyboard: Tab moves focus into the active tab; arrow keys move focus/selection along
  the list (roving tabindex) ŌĆö verified by `NavigatesWithArrows`.
- Closable tabs are removable from the keyboard via Delete/Backspace on the focused tab;
  the inline (x) is `aria-hidden` to avoid nesting an interactive control inside the tab
  button ŌĆö exercised by `Editable`.
- axe runs on every story (`a11y: { test: 'error' }`), so name/role/contrast rules are
  gated in CI.

## Do's and Don'ts

- Ō£ģ Do keep tab labels short and the tab order stable.
- Ō£ģ Do use `closable` only when removing a tab is actually supported.
- ŌØī Don't use Tabs for ordered, multi-step flows ŌĆö use a stepper.
- ŌØī Don't bury content the user always needs behind a tab.
