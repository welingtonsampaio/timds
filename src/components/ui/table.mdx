# Table ¬∑ `Stable`

Displays tabular data in rows and columns, built on native HTML table elements.

## Overview

Use a Table to present structured data ‚Äî records with the same set of attributes ‚Äî so
people can scan and compare across rows. It is a thin, styled wrapper over the native
`<table>` semantics: you compose the parts yourself, keeping full control over content
and accessibility. For sorting, filtering, pagination or column visibility, drive the
markup with a headless table library (e.g. TanStack Table) and render its output through
these parts.

## Anatomy

- **Table** ‚Äî the scroll container plus the `<table>` element; carries the base styling.
- **TableCaption** ‚Äî an optional `<caption>` that names the table for assistive tech.
- **TableHeader** ‚Äî the `<thead>` wrapping the header row(s).
- **TableHead** ‚Äî a `<th>` column header cell.
- **TableBody** ‚Äî the `<tbody>` wrapping the data rows.
- **TableRow** ‚Äî a `<tr>`; mark a selected row with `data-state="selected"`.
- **TableCell** ‚Äî a `<td>` data cell.
- **TableGroupRow** ‚Äî a full-width `<th scope="rowgroup">` row that labels a group of
  rows (e.g. "Overdue", "To pay", "Paid").
- **TableFooter** ‚Äî an optional `<tfoot>` for totals or summaries.

## When to use

- Use to display homogeneous records with shared columns (invoices, users, transactions).
- Use when people need to compare values across rows or scan a column.
- Use the footer for column totals or aggregate summaries.
- Use `TableGroupRow` to split rows into labelled sections (status, category, date) ‚Äî the
  classic finance/CRM case (overdue / to pay / paid).

## When not to use

- Avoid for layout ‚Äî use CSS grid/flex, not a table, to position unrelated content.
- Avoid for a small set of key/value pairs about a single entity ‚Äî a description list or
  **Item** reads better.
- Avoid hand-rolling sort/filter/pagination state ‚Äî compose with a headless table library
  and feed its rows into these parts.

## Props

Each part forwards the props of its native element (`<table>`, `<thead>`, `<tr>`, ‚Ä¶) and
merges `className` last via `cn`, so any utility is overridable.

<Controls of={Stories.Playground} />

### `Table` grouping prop

- **`columnCount`** (`number`) ‚Äî total number of columns. Propagates `colSpan` to every
  `TableGroupRow` via context, so group rows span the full width without hand-counting.
- **`containerClassName`** (`string`) ‚Äî classes for the scroll container (the `<div>`
  wrapping the `<table>`). Constrain its height here (e.g. `max-h-64 overflow-y-auto`) to
  enable vertical scrolling ‚Äî **required for `TableGroupRow sticky`**, whose `top-0`
  anchors to this container, not to an outer wrapper.

### `TableGroupRow` props

`TableGroupRow` renders a full-width `<tr><th colSpan scope="rowgroup">`. It is purely
presentational ‚Äî you supply already-grouped data; it does not compute groups.

- **`children`** (`ReactNode`) ‚Äî the group label.
- **`count`** (`number`) ‚Äî item count, rendered as `(n)` next to the label.
- **`colSpan`** (`number`, default: `Table`'s `columnCount`) ‚Äî columns to span. Falls back
  to the context; pass it explicitly when the `Table` has no `columnCount`.
- **`sticky`** (`boolean`, default `false`) ‚Äî pins the group header to the top on scroll
  (`position: sticky`).

> Provide either `columnCount` on `Table` **or** `colSpan` on each `TableGroupRow` ‚Äî with
> neither, the header spans a single column.

## Behavior & interaction

These behaviors are guarded by interaction tests in `table.stories.tsx`:

- The composition exposes native table semantics ‚Äî roles `table`, `columnheader`, `row`,
  `cell` with the expected counts ‚Äî `RendersTableSemantics`.
- `TableCaption` renders as a `<caption>` that labels the table ‚Äî `CaptionLabelsTable`.
- Checking a row's checkbox marks that row with `data-state="selected"` ‚Äî `SelectsRow`.
- The header "select all" checkbox selects every row at once ‚Äî `SelectsAllRows`.
- The empty state renders a single full-width cell ‚Äî `RendersEmptyState`.
- `TableGroupRow` renders a `<th scope="rowgroup">` spanning all columns, with the count
  resolved from `Table`'s `columnCount` ‚Äî `GroupHeaderSpansColumns`.
- With no `columnCount`, `colSpan` falls back to the prop ‚Äî `GroupHeaderColSpanFallback`.
- `sticky` marks the group cell with `data-sticky` ‚Äî `GroupHeaderIsSticky`.

> **Sticky requires height on the `Table` container.** `position: sticky` anchors to the
> nearest scroll ancestor; the `Table` already wraps its `<table>` in an
> `overflow-x-auto` container, so the vertical scroll must live there too ‚Äî set
> `containerClassName="max-h-‚Ä¶ overflow-y-auto"`. An outer wrapper with the height won't
> work: the inner container becomes the scroll ancestor but never scrolls, so the header
> never detaches.

> Visual states (the sticky band, group background) are verified by Chromatic, not the
> interaction tests ‚Äî the Storybook test project loads no CSS.

## Code examples

```tsx
import {
  Table,
  TableBody,
  TableCaption,
  TableCell,
  TableHead,
  TableHeader,
  TableRow,
} from 'timds'

<Table>
  <TableCaption>Recent invoices.</TableCaption>
  <TableHeader>
    <TableRow>
      <TableHead>Invoice</TableHead>
      <TableHead className="text-right">Amount</TableHead>
    </TableRow>
  </TableHeader>
  <TableBody>
    {invoices.map((row) => (
      <TableRow key={row.invoice}>
        <TableCell className="font-medium">{row.invoice}</TableCell>
        <TableCell className="text-right">{row.amount}</TableCell>
      </TableRow>
    ))}
  </TableBody>
</Table>
```

## Content & UX writing

- **Column headers** ‚Äî short noun phrases in sentence case ("Payment method", not "PAY").
- **Numeric columns** ‚Äî right-align values and headers so digits line up for scanning.
- **Caption** ‚Äî a brief, descriptive label of what the table contains, not a sentence.
- **Group labels** ‚Äî short status nouns ("Overdue", "Paid"); show the count as `(n)`.
- **Empty state** ‚Äî a plain, specific message ("No invoices yet."), never a blank table.

## Accessibility

- Built on native `<table>` semantics: assistive tech exposes roles `table`, `row`,
  `columnheader` and `cell` automatically ‚Äî verified by `RendersTableSemantics`.
- Use `TableCaption` to give the table an accessible name ‚Äî verified by
  `CaptionLabelsTable`.
- Group headers use `<th scope="rowgroup">`, the correct scope for a header that labels a
  group of rows (not `colgroup`). Put the count in the visible text ("Overdue (3)") ‚Äî no
  screen reader counts rows on its own.
- Selection controls must carry an accessible name (e.g. `aria-label="Select INV001"`);
  icon-only or checkbox-only cells are unlabeled otherwise.
- axe runs on every story (`a11y: { test: 'error' }`), so name/role/contrast rules are
  gated in CI.

## Do's and Don'ts

- ‚úÖ Do use `TableCaption` (or another labelling strategy) to name the table.
- ‚úÖ Do right-align numeric columns and keep headers in sentence case.
- ‚úÖ Do show an explicit empty state instead of an empty body.
- ‚úÖ Do set `columnCount` on `Table` so group rows span every column automatically.
- ‚ùå Don't use a Table for visual layout of unrelated content.
- ‚ùå Don't ship selection checkboxes without an `aria-label`.

## Related components & patterns

- **Checkbox** ‚Äî composes into a selection column, per row and a header "select all".
- **Pagination** ‚Äî pairs with a Table to page through large datasets.
- **Item** ‚Äî prefer it for a single entity's attributes rather than a one-row table.
