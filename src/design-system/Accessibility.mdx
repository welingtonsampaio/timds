# Accessibility

Accessibility in timds is a **build-time guarantee**, not a manual review. Every story —
including the component gallery and the **Examples** — renders in a real Chromium
browser and is scanned with [axe](https://github.com/dequelabs/axe-core). A violation
**fails the test**.

```ts
// .storybook/preview.tsx
a11y: { test: 'error' }
```

That single setting is why the examples in this library are written the way they are:
labelled inputs, named icon-only buttons, alt text on avatars, and AA-compliant color.

## Color contrast (WCAG AA)

Text and icons meet a contrast ratio of **≥ 4.5:1** against their background (≥ 3:1 for
large text) in **both** light and dark modes. This is enforced because the Storybook
test runner loads the real Tailwind CSS, so axe measures contrast on the actual styled
DOM.

The key rule that follows from this:

> **Vivid fill tokens are not text colors.** `--success` / `--warning` / `--info` /
> `--primary` are tuned as **fills** (dark text on top). To color a glyph or a link on a
> light surface, use the darker `*-text` variant — `text-success-text`,
> `text-warning-text`, `text-info-text`, `text-primary-text`, `text-destructive-text` —
> which passes AA. See **Theming → Fill tokens vs. text tokens**.

## Keyboard & focus

- Every interactive element is reachable and operable by keyboard.
- A consistent focus ring (the shared `focusRing` preset, built on the `--ring` token)
  marks the focused control — visible in both themes.
- Composite widgets (Tabs, Select, Sidebar, menus) follow the WAI-ARIA APG patterns
  from their underlying primitives (Radix, and Ariakit for `Select`).

## Opting out responsibly

When a specific axe rule genuinely cannot apply to a story, disable that **single rule**
with a documented reason — never the whole check:

```tsx
parameters: {
  a11y: {
    config: {
      rules: [{ id: 'color-contrast', enabled: false }], // motivo: …
    },
  },
}
```

Prefer fixing the markup. A disabled rule is a debt, and it should be rare.
