# Button · `Stable`

Triggers an action or event — the primary way users commit to something in the UI.

## Overview

Use a Button to let people take an action: submit a form, confirm a choice, open a
dialog. It is not for navigation between pages (use a Link for that). Each view should
have a single primary Button for its main call to action.

<Canvas of={Stories.Default} />

## Anatomy

- **Container** — the interactive surface; carries the variant styling and the hit
  target.
- **Label** — a verb-first action ("Save", "Delete account").
- **Icon (optional)** — a leading or trailing icon reinforcing the action.
- **Spinner** — replaces the icon while `loading` is true.

## When to use

- Use to commit an action: submit, confirm, save, delete.
- Use the **default** variant for the single most important action in a view.
- Use **secondary** / **ghost** for supporting actions.

## When not to use

- Avoid for navigation to another page — pass `href` only when a button-styled link is
  genuinely intended, otherwise use a Link.
- Avoid more than one primary Button in the same group — it dilutes the call to action.

## Behavior & interaction

These behaviors are guarded by interaction tests in `button.stories.tsx`:

- Activation calls `onClick` once — `ClicksOnce`.
- While loading, the button is disabled, exposes `aria-busy` and ignores clicks —
  `LoadingBlocksClicks`.
- The button is keyboard-focusable — `FocusesWithKeyboard`.

## Code examples

import { Button } from 'timds'

## Content & UX writing

- **Labels** — verb-first, ideally one or two words ("Save", "Add member"). Avoid vague
  labels like "OK" when a specific verb fits.
- **No terminal punctuation** in labels.
- **Icon-only** buttons must carry an accessible name via `aria-label`.

## Accessibility

- Reachable and operable by keyboard: Tab to focus, Enter/Space to activate — verified by
  `FocusesWithKeyboard`.
- Exposes the native `button` role; `aria-busy="true"` reflects the loading state —
  verified by `LoadingBlocksClicks`.
- axe runs on every story (`a11y: { test: 'error' }`), so contrast/name/role rules are
  gated in CI.

## Do's and Don'ts

- ✅ Do use one primary Button per view for the main action.
- ✅ Do show a loading state for async actions to prevent double submits.
- ❌ Don't disable a Button without making the reason discoverable nearby.
- ❌ Don't ship an icon-only Button without an `aria-label`.

## Related components & patterns

- **ButtonGroup** — composes multiple related actions; keep at most one primary inside it.
- **Spinner** — the loading indicator the `loading` flag renders in the icon slot.
